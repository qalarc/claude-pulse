How Claude Pulse works

Claude Pulse is a local-first usage monitor. It watches your own Claude API traffic and visualizes it — it never sends your prompts, completions, or credentials anywhere. Everything stays on your machine.

1 · How it collects data

A tiny zero-dependency collector proxy sits between your Claude tools and Anthropic. You point a tool at it with ANTHROPIC_BASE_URL=http://localhost:8787; it transparently forwards every request to api.anthropic.com and streams the response straight back, untouched.

Claude tools → collector proxy → api.anthropic.com

As each response passes through, the collector records metadata only to an append-only log (~/.local/share/claude-pulse/usage.jsonl):

  • token counts from the response usage object (input / output / cache)
  • the anthropic-ratelimit-* headers (utilization + reset times)
  • HTTP status, 429s, and timing

It never writes prompt text, completion text, or your auth token. The desktop app and the web dashboard simply read this log and aggregate it into the gauge, the per-minute graph, and the daily calendar.

2 · Where the live limits come from

The 5-hour (“current session”) and 7-day (“weekly, all models”) windows — and their reset countdowns — are the real values Anthropic returns to you. Every response that passes through the collector carries anthropic-ratelimit-unified-* headers; the collector logs them, so the limits come straight from your own traffic — no extra software required. Just keep using Claude through the collector and the bars stay current.

Optional: if you happen to run a qalcode2 / opencode server locally, Pulse will also read its /ratelimit endpoint to fill the windows in before your first proxied request and to show your plan label — but this is a bonus, not a requirement. With only an API key (no Pro/Max OAuth) the unified windows don’t exist, so those bars stay empty by design while your token/request counts still graph normally.

3 · What happens when you swap subscriptions

The limits Pulse displays are tied to whichever account / plan made the request, because they come from that request’s response headers:

  • Upgrading (e.g. Pro → Max, or Max 5× → Max 20×): your ceilings rise, so the same workload reads as a lower utilization percentage. The 5h/7d bars drop and the “Max 20×” plan label updates on the next request after the new plan is active.
  • Downgrading: ceilings shrink — the same usage now reads as a higher percentage, and you’ll hit 429s sooner. Pulse will start showing the red rate-limit markers earlier.
  • Switching account / auth (API key ↔ OAuth): the metric model changes. OAuth/subscription shows unified 5h/7d windows; API-key billing shows tokens/requests-remaining instead. Pulse renders whichever set the headers provide.
  • History is preserved: the graph and calendar keep showing past token usage across the swap (it’s your raw traffic). Only the percentage readings re-baseline to the new plan’s limits — the absolute tokens you sent don’t change retroactively.

Tip: right after switching plans, make one request through the collector so a fresh set of rate-limit headers is captured and the windows + plan label update.

4 · Privacy & safety

  • Metadata only — never prompts, completions, or tokens.
  • Local-first — no account, no cloud, no telemetry.
  • Transparent proxy — requests/responses stream through untouched.
  • Read-your-own-account — all limit data comes from headers Anthropic returns to you. No probing or brute-forcing.
