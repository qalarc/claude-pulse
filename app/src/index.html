How Claude Pulse works

Claude Pulse is a local-first usage monitor. It watches your own Claude API traffic and visualizes it — it never sends your prompts, completions, or credentials anywhere. Everything stays on your machine.

1 · How it collects data — two sources

Claude Pulse gets metadata only (never prompts, completions, or your auth token) from whichever of these is available, and writes it to an append-only log (~/.local/share/claude-pulse/usage.jsonl) that the app and dashboard read:

A · A live opencode / qalcode2 server (primary). If one is running, Pulse polls its HTTP API (GET /session/:id/message) for each assistant message's real token counts, and GET /ratelimit for the 5h/7d windows. This is how your tokens/min populates with no proxy and no setup. If several servers are running (old + current sessions), Pulse always picks the freshest one (newest snapshot) so you never see another session's stale data.

opencode / qalcode2 → /session, /ratelimit → usage.jsonl

B · The collector proxy (any tool). For tools that don't expose an API, point them at the proxy with ANTHROPIC_BASE_URL=http://localhost:8787. It transparently forwards every request to api.anthropic.com, streams the response back untouched, and records the usage object + anthropic-ratelimit-* headers + status/timing.

Claude tools → collector proxy :8787 → api.anthropic.com

2 · What each number means

  • Tokens / min graphs input + output per minute. Cache reads are excluded (they're cheap context re-reads; Anthropic excludes them from rate-limit accounting too) — so the headline number is "new" token throughput, not raw context size.
  • 5-hour / 7-day windows are the real utilization values Anthropic returns for your account (the anthropic-ratelimit-unified-* headers), with live "resets in" countdowns and your plan label (e.g. Max 20×).
  • Rate-limit (429) marks show where you got throttled, on the graph and as history ticks on the gauge.

With only a plain API key (no Pro/Max OAuth) the unified 5h/7d windows don't exist, so those bars stay empty by design — token/request counts still graph normally.

3 · What happens when you swap subscriptions

The limits Pulse displays are tied to whichever account / plan made the request, because they come from that request’s response headers:

  • Upgrading (e.g. Pro → Max, or Max 5× → Max 20×): your ceilings rise, so the same workload reads as a lower utilization percentage. The 5h/7d bars drop and the “Max 20×” plan label updates on the next request after the new plan is active.
  • Downgrading: ceilings shrink — the same usage now reads as a higher percentage, and you’ll hit 429s sooner. Pulse will start showing the red rate-limit markers earlier.
  • Switching account / auth (API key ↔ OAuth): the metric model changes. OAuth/subscription shows unified 5h/7d windows; API-key billing shows tokens/requests-remaining instead. Pulse renders whichever set the headers provide.
  • History is preserved: the graph and calendar keep showing past token usage across the swap (it’s your raw traffic). Only the percentage readings re-baseline to the new plan’s limits — the absolute tokens you sent don’t change retroactively.

Tip: right after switching plans, make one request through the collector so a fresh set of rate-limit headers is captured and the windows + plan label update.

4 · Privacy & safety

  • Metadata only — never prompts, completions, or tokens.
  • Local-first — no account, no cloud, no telemetry.
  • Transparent proxy — requests/responses stream through untouched.
  • Read-your-own-account — all limit data comes from headers Anthropic returns to you. No probing or brute-forcing.
